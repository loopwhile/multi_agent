# Escalation Protocol

## 목적

역할 간 판단 충돌, 스코프 대립, 기술적 불일치가 발생했을 때의 **에스컬레이션 규칙**을 정의한다.  
모든 에이전트는 이 프로토콜을 따라 충돌을 해결한다.

---

## 에스컬레이션 레벨

| 레벨 | 설명 | 해결 주체 |
|------|------|-----------|
| **L1 — 역할 간 직접 해결** | 두 역할이 직접 근거를 교환하여 합의 | 당사자 역할 |
| **L2 — another_me 중재** | L1에서 합의 불가 시 another_me가 최종 판정 | another_me |
| **L3 — 사용자 판단** | another_me도 판단이 어려운 경우 사용자에게 에스컬레이션 | 사용자 |

---

## 에스컬레이션 트리거

| 상황 | 레벨 | 상대 역할 |
|------|------|-----------|
| 기능 범위 vs 기술 실현성 대립 | L1 → L2 | product_scope ↔ architect |
| 설계 결함 발견 (구현 중) | L1 | builder → architect |
| 검증 판정 불일치 | L1 → L2 | builder ↔ quality |
| 배포 실패 / 긴급 롤백 | L2 | operations → another_me |
| 스코프 변경이 아키텍처에 영향 | L1 | scope → architect |
| 품질 기준 자체에 의문 | L2 | quality → another_me |
| 역할 경계 모호 | L2 | 어떤 역할이든 → another_me |
| 전략적 방향 판단 | L3 | another_me → 사용자 |

---

## 에스컬레이션 형식

에스컬레이션 시 다음 형식을 사용한다:

```markdown
## Escalation Request

### 발신 역할
{에스컬레이션을 요청하는 역할}

### 수신 역할
{판단을 요청받는 역할}

### 레벨
L1 / L2 / L3

### 대상 Work Order
{관련 WO-ID, 없으면 N/A}

### 문제 설명
{충돌/대립의 구체적 내용}

### 발신 측 입장 + 근거
{자신의 판단과 그 근거}

### 상대 측 입장 + 근거 (L1에서 L2 전환 시)
{상대 역할의 판단과 그 근거}

### 요청 판단
{무엇을 결정해 주기를 원하는가}
```

---

## 해결 규칙

1. **L1 우선** — 가능한 한 당사자 간 직접 해결. 매 충돌마다 L2로 올리지 않는다
2. **근거 필수** — 에스컬레이션 시 양측 모두 구체적 근거를 제시해야 한다
3. **another_me의 판정은 최종** — L2 판정 후 같은 건으로 재에스컬레이션 금지
4. **판정 기록** — 모든 L2 이상의 판정은 Work Order 비고에 기록
5. **시간 제한** — L1은 1회 교환으로 합의 시도. 합의 불가 시 즉시 L2 전환
6. **사용자 에스컬레이션 최소화** — L3은 전략적 방향이나 비즈니스 판단에만 사용

---

## another_me 중재 시 판단 기준

another_me는 중재 시 다음 순서로 판단한다:

1. 핵심 제품 루프를 보존하는 방향은 무엇인가
2. 장기적으로 더 큰 레버리지를 만드는 방향은 무엇인가
3. 구조적 일관성을 유지하는 방향은 무엇인가
4. 운영 부담이 더 적은 방향은 무엇인가
5. 지금 결정하지 않아도 되는 것은 미루고, 반드시 결정해야 할 것만 판정
